# whisper_ptt Фаза 2: OpenVINO-бэкенд для Intel iGPU (large-v3 на ноуте Honor)

**Дата:** 2026-07-04
**Статус:** дизайн утверждён (брейншторм с Русланом), ждёт ревью спеки перед планом
**Предыдущая спека:** `2026-06-04-portable-exe-cross-platform-design.md` (Фаза 1, задел под этот бэкенд)

## Цель

Быстрое локальное распознавание уровня large-v3 на ноутбуке без NVIDIA:
Honor MagicBook Art 14 — Core Ultra 7 255H (16 потоков), **Intel Arc 140T iGPU**
(до 16 ГБ общей памяти, подтверждено по реестру), NPU, 32 ГБ RAM, Windows 11.

**Критерий успеха:** задержка «отпустил кнопку → текст вставлен» ≤ 3–4 с для
короткой фразы (3–5 с аудио).

**Приоритет модели:** если large-v3 укладывается в критерий на iGPU — она дефолт;
иначе дефолт large-v3-turbo (качество на русском сопоставимо, скорость в разы выше),
а large-v3 остаётся доступной в настройках.

## Скоуп

Только локальный инференс (OpenVINO-бэкенд + CPU-замеры как база сравнения).
**API-бэкенд (облако) — вне скоупа**, отдельная следующая итерация.

## Решения, принятые на брейншторме

- **Путь A с подстраховкой B:** основной путь — OpenVINO GenAI на Arc 140T;
  до интеграции — быстрые замеры CPU (small/turbo/large-v3 через существующий
  CTranslate2-бэкенд) как честная база и вечный запасной режим.
- **Сначала прототип-бенч, потом интеграция.** Ворота решения по замерам
  (см. «Порядок работ»), чтобы не строить интеграцию на непроверенной скорости.
- **Готовые модели, без конвертации.** С HuggingFace, организация OpenVINO:
  `OpenVINO/whisper-large-v3-int8-ov` (~1.6 ГБ), `OpenVINO/whisper-large-v3-turbo-int8-ov`
  (~0.9 ГБ). PyTorch/optimum на машину пользователя не ставятся.
- **whisper.cpp/Vulkan отклонён:** на Intel обычно медленнее OpenVINO (без XMX),
  лишний рантайм и формат моделей, слабее ложится на нашу архитектуру.
- **NPU — экспериментальная опция** (значение устройства в конфиге), не дефолт.
  OpenVINO GenAI поддерживает NPU через `STATIC_PIPELINE`; пробуем после GPU-замеров,
  в дефолтную логику авто-выбора NPU не входит.

Факты, подтверждённые по докам (context7) и HF API:

- `openvino_genai.WhisperPipeline(model_dir, device)` — device `"CPU"|"GPU"|"NPU"`;
  `generate(raw_speech: list[float] 16 кГц, config)`; `config.language = "<|ru|>"`,
  `task="transcribe"`, `return_timestamps=True` → `result.chunks`
  (`start_ts`, `end_ts`, `text`); поддерживаются `initial_prompt` и `hotwords`.
- Готовые int8/int4/fp16 OV-модели whisper (включая large-v3 и large-v3-turbo)
  опубликованы организацией OpenVINO на HF (проверено по API 2026-07-04).
- `optimum-cli export openvino` существует как запасной путь конвертации
  (на этой машине не используется).

## Архитектура

### A. OpenVINOBackend (backends.py)

Заглушка Фазы 1 наполняется реализацией:

- `load()`: `openvino_genai.WhisperPipeline(model_dir, device, CACHE_DIR=<data_dir>/ov_cache)`.
  Кэш компиляции обязателен: первая компиляция под конкретный GPU занимает
  десятки секунд, с кэшем тёплый старт быстрый.
- `transcribe(audio, cfg)`:
  1. **VAD-шаг** (при `cfg.vad_filter`): переиспользуем Silero-VAD из faster-whisper
     (`faster_whisper.vad.get_speech_timestamps` + `collect_chunks`) — faster-whisper
     в любом случае остаётся зависимостью (CPU/CUDA-путь). Нет речи → сразу `([], info)`.
  2. `pipe.generate(...)` с `language` (маппинг `"ru"` → `"<|ru|>"`; пустой язык →
     авто-детект пайплайна), `task="transcribe"`, `return_timestamps=True`,
     `initial_prompt=cfg.initial_prompt`.
  3. **Адаптер результата** (чистая функция): `result.chunks` → список сегментов
     с полями `.text`, `.start`, `.end`, `.compression_ratio=0.0`; `info` c полями
     `.language` (из конфига либо `""`), `.language_probability=1.0`, `.duration`.
     Контракт `(segments, info)` тот же, что у CTranslate2Backend — dictate.py,
     postprocess.py и UI не меняются.
- `device_label`: `"Intel GPU (OpenVINO)"` / `"Intel NPU (OpenVINO)"`.
- `model_id`: HF-repo OV-модели (для докачки через model_store).

Параметры CT2, не имеющие прямого аналога в GenAI (`beam_size`,
`no_repeat_ngram_size`, `condition_on_previous_text`), маппятся best-effort:
что поддерживается `WhisperGenerationConfig` — передаём, остальное игнорируем.
Точный маппинг уточняется на прототипе; главные защиты от галлюцинаций
(VAD, блок-лист, дедуп, compression_ratio для CT2-пути) сохраняются.

**Деградация (осознанная):** `min_language_probability` в OV-пути не работает
(пайплайн не отдаёт уверенность языка; в адаптере всегда 1.0). Дефолт параметра
и так `0.0` (выключен). Помечаем в README и подсказке UI.

### B. Выбор модели для iGPU

Маппинг имён моделей на OV-репозитории (таблица в backends.py):
`large-v3` → `OpenVINO/whisper-large-v3-int8-ov`;
`large-v3-turbo`/`turbo` → `OpenVINO/whisper-large-v3-turbo-int8-ov`;
`small`/`medium`/… → соответствующие `OpenVINO/whisper-<имя>-int8-ov`.
Модель без OV-аналога при `igpu` → понятная ошибка с советом сменить модель.

`compute_type` в OV-пути игнорируется (квантование зашито в скачанную модель).

Дефолтная модель на iGPU определяется воротами бенча (см. «Порядок работ»)
и фиксируется константой `IGPU_DEFAULT_MODEL` (аналог `CPU_FALLBACK_MODEL`):
критерий — large-v3 ≤ 3–4 с на фразу 3–5 с → `large-v3`, иначе `large-v3-turbo`.
Понижение тяжёлых моделей (HEAVY_MODELS → small) на iGPU **не** применяется —
оно остаётся только для чистого CPU.

### C. Конфиг и маршрутизация (config.py, backends.py)

- `device` расширяется: `auto / cuda / cpu / igpu / npu / api`.
- Авто-детект (`resolve_runtime` + новая проба):
  `auto` → есть CUDA → `cuda`; иначе есть Intel GPU (`"GPU" in openvino.Core().available_devices`,
  лениво, в try/except — любой сбой = нет) → `igpu`; иначе → `cpu`.
- Маршрутизация `select_backend`: `igpu`/`npu` → `OpenVINOBackend` (устройство
  пайплайна: `igpu` → `"GPU"`, `npu` → `"NPU"`); остальное — как сейчас.

### D. Хранилище моделей (model_store.py)

- Скачивание OV-моделей: `huggingface_hub.snapshot_download(repo_id, local_dir=...)`
  (huggingface_hub уже установлен как зависимость faster-whisper).
- Раскладка та же: `<data_dir>/models/<repo_id с заменой '/' на '_'>`.
- `is_cached`: для CT2-моделей — `model.bin` (как сейчас); для OV-моделей —
  файл-маркер `.download_complete`, записываемый после успешного snapshot_download
  (состав файлов OV-репо может меняться, маркер надёжнее перечня).

### E. UI (gui.py)

- Комбобокс «Устройство»: пункты «Intel GPU (OpenVINO)» и «Intel NPU (эксперимент)»;
  видны всегда, как и «GPU (CUDA)» (устройства может не быть — тогда при загрузке
  фолбэк с уведомлением, единое поведение).
- Статус показывает `device_label` реального бэкенда (already works через Фазу 1).
- При авто-фолбэке `igpu → cpu` (нет драйвера/устройства) — разовое уведомление,
  как для CUDA→CPU.

### F. Зависимости

- Новая: `openvino-genai` (тянет `openvino`, `openvino-tokenizers`; ~200 МБ, без torch).
- `librosa` НЕ нужна (аудио уже numpy float32 16 кГц из sounddevice; `.tolist()`).
- requirements.txt / requirements.lock.txt обновляются.

## Порядок работ

1. **venv на ноуте** + запуск как есть (auto → cpu → small): смоук «приложение живо».
2. **Бенч-прототип** (`bench_backends.py`, остаётся в репо как диагностика):
   - режим записи: надиктовать 3–5 реальных фраз (короткая/средняя/длинная), сохранить WAV;
   - режим прогона: одни и те же WAV через матрицу
     {CPU-CT2: small, turbo, large-v3} × {OV-GPU: turbo, large-v3};
   - вывод: markdown-таблица (время, RTF, текст) — сравнение скорости и качества глазами.
   - Требует установленного `openvino-genai` и скачанных OV-моделей (докачка в бенче).
3. **Ворота решения:** по таблице фиксируем `IGPU_DEFAULT_MODEL` (критерий выше)
   и подтверждаем сам путь OpenVINO (если Arc внезапно медленнее CPU — стоп,
   пересматриваем: остаёмся на CPU+turbo, дизайн-ревизия).
4. **Интеграция:** backends/config/model_store/gui + тесты (см. ниже).
5. **Ручной смоук на ноуте:** полный цикл диктовки на iGPU, тёплый/холодный старт,
   фолбэк при выдернутом драйвере не проверяем (нереалистично) — фолбэк покрыт юнитом.
6. **Документация:** README (новые значения device, таблица конфига, заметка про
   min_language_probability), при необходимости — заметка про драйвер Arc.

## Обработка ошибок

- Нет Intel GPU / старый драйвер / сбой компиляции OV → при `auto` фолбэк на CPU
  с уведомлением; при явном `igpu` — понятное сообщение в UI, приложение живо.
- Обрыв скачивания OV-модели → маркер не записан → следующая попытка докачает
  (snapshot_download докачивает недостающие файлы); сообщение в UI как сейчас.
- NPU не скомпилировал модель → сообщение «NPU недоступен, выбери iGPU/CPU» (эксперимент).

## Тестирование

Без железа (юниты, в CI/на любой машине):

- адаптер chunks→segments/info: пустой результат, несколько чанков, поля контракта;
- маршрутизация: `resolve_runtime`/`select_backend` с моками проб
  (cuda=нет, ov=есть → igpu; оба нет → cpu; явный igpu без пробы);
- model_store: `is_cached` для OV-маркера; путь с заменой `/`;
- VAD-шаг: мок get_speech_timestamps (нет речи → пустой результат, есть → вызывается generate).

На железе (вручную): бенч-таблица (шаг 2), полный цикл диктовки (шаг 5).

Существующие тесты (test_backends, test_postprocess, test_transcribe_pipeline,
smoke_gui) остаются зелёными; новые случаи маршрутизации добавляются рядом
с текущими в test_backends.py, CPU/CUDA-путь поведения не меняет.

## Риски

- **Скорость Arc 140T неизвестна до бенча** — главный риск, снимается шагом 2
  (прототип до интеграции). Запасной план встроен: CPU + turbo.
- **Свежесть драйвера Intel Arc** — проверяем в начале; обновление драйвера
  может потребовать ручных действий пользователя.
- **Внутренний API faster_whisper.vad** — не публичный контракт; версия
  запинена в requirements.lock, при апгрейде faster-whisper тест VAD-шага
  подсветит поломку.
- **beam search в GenAI** может отсутствовать/отличаться — качество проверяем
  глазами на бенче (шаг 2), решение фиксируем там же.
- **Дачный интернет:** модели 0.9–1.6 ГБ; докачка возобновляема (snapshot_download).
- **Холодная компиляция OV на GPU** — разово десятки секунд; лечится CACHE_DIR
  (тёплый старт). В UI на время компиляции показывается состояние «загружаю модель».

## Вне скоупа

API-бэкенд (следующая итерация), конвертация собственных моделей через optimum
(пользуемся готовыми), NPU как дефолт (только экспериментальная опция),
изменение CPU-пути и Фазы 1.
